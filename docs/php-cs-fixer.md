# PHP-CS-Fixer Configuration Guide

PHP-CS-Fixer configuration generated by the `/setup-toolkit-php-cs-fixer` skill.

## Configuration

```php
<?php

declare(strict_types=1);

$finder = (new PhpCsFixer\Finder())
    ->in(__DIR__)
    ->exclude('vendor');

return (new PhpCsFixer\Config())
    ->setRiskyAllowed(true)
    ->setRules([
        '@PSR12' => true,
        'declare_strict_types' => true,
        'strict_param' => true,
        'strict_comparison' => true,
        'array_syntax' => ['syntax' => 'short'],
        'no_unused_imports' => true,
        'ordered_imports' => ['sort_algorithm' => 'alpha'],
        'single_quote' => true,
        'trailing_comma_in_multiline' => true,
        'no_empty_statement' => true,
        'no_superfluous_elseif' => true,
        'no_useless_else' => true,
        'void_return' => true,
        'no_alias_functions' => true,
        'no_mixed_echo_print' => ['use' => 'echo'],
        'global_namespace_import' => [
            'import_classes' => true,
            'import_constants' => true,
            'import_functions' => true,
        ],
        'fully_qualified_strict_types' => true,
    ])
    ->setFinder($finder);
```

## Explanation

### `@PSR12`

Applies PSR-12 coding standard as the baseline.

**Why:** PSR-12 is the standard coding convention in the PHP community, adopted by nearly every PHP project. It serves as the baseline for unifying the formatting of AI-generated code.

### `declare_strict_types`

Forces `declare(strict_types=1)` in every PHP file.

**Why:** PHP has type coercion enabled by default—`"1"` is silently converted to `1`. Declaring `strict_types` turns type mismatches into `TypeError` exceptions. When an AI agent generates a file without the `strict_types` declaration, this rule auto-adds it.

### `strict_param`

Forces the strict comparison flag (`true` as the 3rd argument) on `in_array()`, `array_search()`, `array_keys()`, etc.

**Why:** These PHP functions use loose comparison by default—`in_array(0, ['foo'])` returns `true`. The 3rd argument `true` enables strict comparison, but it is easy to omit. When an AI agent omits this flag, it is auto-added.

### `strict_comparison`

Auto-fixes `==` to `===` and `!=` to `!==`.

**Why:** PHP's `==` performs type juggling—`0 == "foo"` was `true` in PHP 7 and earlier—and there is no use case for it. AI agents have generated `==`, so this setting auto-fixes it to `===`.

### `array_syntax` (`short`)

Normalizes array syntax from `array()` to `[]`.

**Why:** `[]` has been available since PHP 5.4, so there is no reason to use `array()`. AI agents mix `array()` and `[]`, so this unifies them.

### `no_unused_imports`

Auto-removes unused `use` statements.

**Why:** Unused `use` statements are unnecessary clutter. AI agents leave behind `use` statements that are no longer referenced after refactoring, so they are auto-removed.

### `ordered_imports` (`alpha`)

Sorts `use` statements alphabetically.

**Why:** Alphabetical ordering is deterministic—AI agents can always compute the correct insertion position. Other algorithms like `length` cause reordering when names change, producing unnecessary diffs. AI agents add new `use` statements at inconsistent positions, so `alpha` normalizes them.

### `single_quote`

Normalizes string literals to single quotes.

**Why:** There is no reason to use double quotes for strings without variable interpolation. Single quotes make intent explicit.

### `trailing_comma_in_multiline`

Adds trailing commas to multiline arrays, argument lists, and parameter lists.

**Why:** Trailing commas reduce diffs to a single line when adding or removing elements. This reduces diff noise when AI agents add elements.

### `no_empty_statement`

Removes empty statements (standalone semicolons, etc.).

**Why:** Empty statements are leftovers from code deletion and serve no purpose. AI agents sometimes leave semicolons behind when deleting code, so they are auto-removed.

### `no_superfluous_elseif` / `no_useless_else`

Removes unnecessary `elseif` and `else` blocks.

**Why:** When an `if` block ends with `return`, `throw`, `continue`, etc., the following `else` is unnecessary and only deepens nesting. AI agents add unnecessary `else` blocks, so they are auto-removed.

### `void_return`

Auto-adds `: void` return type to methods with no return value.

**Why:** Explicit `: void` enables PHPStan to infer return types accurately. When an AI agent omits the return type, it is auto-added.

### `no_alias_functions`

Normalizes alias functions (`sizeof` → `count`, `join` → `implode`, etc.) to their canonical names.

**Why:** Alias functions like `sizeof`/`join` are identical to their canonical names (`count`/`implode`); there is only reason to unify them. AI agents mix aliases and canonical names, so they are normalized.

### `no_mixed_echo_print` (`echo`)

Normalizes `print` to `echo`.

**Why:** `echo` and `print` are functionally equivalent; there is only reason to unify them. Standardized on `echo`.

### `global_namespace_import`

Imports all global namespace classes, functions, and constants via `use` statements.

**Why:** Uses explicit `use function strlen;` instead of fully-qualified `\strlen()`. This makes dependencies visible at a glance in the `use` block at the top of the file.

### `fully_qualified_strict_types`

Uses imported short names instead of fully-qualified names in type declarations.

**Why:** Works in conjunction with `global_namespace_import` to unify type references as short names. Prevents AI agents from mixing fully-qualified names like `\DateTime` with short names like `DateTime`.

### `setRiskyAllowed(true)`

Allows "risky" fixer rules to be applied.

**Why:** `declare_strict_types`, `strict_param`, `strict_comparison`, `void_return`, and `no_alias_functions` are all classified as "risky" because they may change existing code behavior. However, every one of these changes is a correction in the right direction, so they are allowed.
