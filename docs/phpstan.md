# PHPStan Configuration Guide

PHPStan configuration generated by the `/setup-toolkit-phpstan` skill.

## Configuration

```neon
includes:
    - vendor/phpstan/phpstan-strict-rules/rules.neon
    - vendor/k-kinzal/php-ai-toolkit/extension.neon
    - vendor/k-kinzal/php-ai-toolkit/error-formatter.neon

parameters:
    level: max
    paths:
        - src
        - tests
```

## Explanation

### `level: max`

Sets PHPStan analysis level to the maximum.

**Why:** Raising the PHPStan level mid-project produces a flood of errors, making gradual migration difficult. Starting at max catches type inconsistencies from the very beginning. When an AI agent generates type-mismatched code, it gets immediate feedback and is forced to fix it.

### `phpstan-strict-rules`

Adds the official PHPStan strict rule set.

**Why:** PHP permits patterns that are technically valid but practically bug-prone: type juggling with `==`, unused parameters, unsafe casts, etc. Strict rules detect these statically. When an AI agent generates these patterns, they are reported as errors.

### `extension.neon` (php-ai-toolkit rules)

Enables custom rules that detect anti-patterns from AI code generation.

**Why:** Standard PHPStan rules do not cover project-specific constraints such as test class structure and comment conventions. See [phpstan-rules.md](phpstan-rules.md) for details.

### `error-formatter.neon` (AI error formatter)

Formats PHPStan error output for AI agent consumption.

**Why:** PHPStan's default error output is designed for humans. AI agents sometimes fail to interpret the context—file paths, line numbers, fix hints—correctly. The AI formatter structures this output to improve the accuracy of AI-driven fixes.

### `paths: [src, tests]`

Includes both `src` and `tests` in the analysis scope.

**Why:** Since AI agents also generate test code, the same static analysis must apply to tests. Type errors in test code undermine test reliability.
